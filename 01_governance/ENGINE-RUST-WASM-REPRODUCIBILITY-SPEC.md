# ENGINE RUST+WASM REPRODUCIBILITY SPEC

## 1. PURPOSE

Niniejszy dokument stanowi techniczny kontrakt (technical specification) dla deterministycznego silnika behavioralnego SILENCE.OBJECTS, implementowanego w języku Rust i kompilowanego do WebAssembly (WASM). Specyfikacja definiuje: model danych, kontrakt determinizmu, propagację seed, interfejs FFI do TypeScript, pipeline reproducible builds, podpisywanie binarne oraz kryteria równoważności (equivalence testing) między Rust natywnym, WASM i CPU fallback.

Dokument jest SSoT dla wszystkich decyzji architektonicznych dotyczących silnika. Każda zmiana wymaga nowego wpisu w EffectLog i review przez engine-lead.

## 2. SCOPE

### In Scope (MVP)
- Deterministyczny scheduler oparty na phi-derived timing windows
- Rust core z jedną ścieżką wykonania (CPU-only, single-threaded, batch)
- Kompilacja do WASM z target `wasm32-unknown-unknown`
- FFI TypeScript przez `wasm-bindgen` + generated `.d.ts`
- Reproducible Docker builds z pinned base image (`rust:1.78-slim-bookworm`)
- Cryptographic signing WASM (SHA-256 hash + Ed25519 signature)
- Equivalence test runner: Rust natywny vs WASM vs CPU fallback (TypeScript)
- No GPU, no SIMD, no multi-region, no ML model

### Out of Scope (MVP)
- GPU acceleration (CUDA, Metal, Vulkan)
- SIMD vectorization (AVX, NEON)
- Streaming / real-time inference
- Multi-region deployment
- Machine learning models (neural networks, statistical inference)
- Blockchain anchoring (planowane poza MVP)

### Dependencies
- MATH_CORE (`00_identity/MATH_CORE.ts`) — źródło stałych phi
- COMP-01 Annex IV Section 2.2 — opis algorytmu deterministycznego
- S11-01 v2.1 — terminologia (brak terminów medycznych/diagnostycznych w kodzie)
- RULE-DOM-001 — silnik jest Open-Core (`04_packages/@silence/engine/` lub osobny crate), nie EE

## 3. DATA MODEL

### 3.1 Core Structs (Rust)

```rust
/// Unique user identifier — UUIDv4, never PII
pub type UserPseudonym = [u8; 16];

/// Attention profile depth — self-reported, categorical 1-5
#[repr(u8)]
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum AttentionDepth {
    Surface = 1,
    Shallow = 2,
    Moderate = 3,
    Deep = 4,
    Flow = 5,
}

/// Input to the deterministic scheduler
#[derive(Clone, Debug)]
pub struct InterventionInput {
    pub user_pseudonym: UserPseudonym,
    pub timestamp_ms: u64,
    pub attention_depth: AttentionDepth,
    pub last_intervention_ms: Option<u64>,
    pub entropy: [u8; 8], // additional entropy for hash collision resistance
}

/// Single scheduled intervention
#[derive(Clone, Debug)]
pub struct InterventionSlot {
    pub scheduled_ms: u64,
    pub priority: u8, // 1-3
    pub intervention_id: [u8; 16], // UUIDv4
}

/// Complete schedule output
#[derive(Clone, Debug)]
pub struct ScheduleOutput {
    pub slots: Vec<InterventionSlot>,
    pub input_hash: [u8; 32],  // SHA-256
    pub seed: [u8; 32],        // SHA-256(input_hash)
    pub output_hash: [u8; 32], // SHA-256(seed + serialized slots)
}
```

## 4. DETERMINISM CONTRACT

### 4.1 Definicja determinizmu
n
Dla każdej pary inputów `I1, I2`:
- Jeśli `I1 == I2` (bitwise equality wszystkich pól), to `output_hash(I1) == output_hash(I2)`
- Jeśli `I1 != I2`, to `Pr[collision] < 2^-256` (właściwość SHA-256)

### 4.2 Źródła nondeterminizmu — explicit zakazane
| Źródło | Status | Uzasadnienie |
|---|---|---|
| System time (wall clock) | ZAKAZANE | `timestamp_ms` jest inputem, nie pobierany wewnątrz engine |
| Random number generator (OS entropy) | ZAKAZANE | `entropy` jest explicit inputem; brak `rand::thread_rng()` |
| Floating-point operations zależne od architektury | ZAKAZANE | Używamy wyłącznie fixed-point integer arithmetic na timestampach |
| Thread scheduling / async runtime | ZAKAZANE | Single-threaded, synchroniczny batch |
| Uninitialized memory | ZAKAZANE | Rust memory safety + explicit zeroing |
| Iterator order zależny od adresów pamięci | ZAKAZANE | `BTreeMap` lub sorted `Vec` zamiast `HashMap` |

### 4.3 Dozwolone źródła zmienności (poza kontraktem determinizmu)
- Kolejność slotów w `ScheduleOutput.slots` — zawsze sortowana rosnąco po `scheduled_ms`
- Format serializacji — zawsze little-endian fixed-width

## 5. SEED PROPAGATION

### 5.1 Hash chain

```
input_hash  = SHA-256( user_pseudonym || timestamp_ms (u64 LE) || attention_depth (u8) || last_intervention_ms (Option<u64 LE>) || entropy (8 bytes) )
seed        = SHA-256( input_hash )
output_hash = SHA-256( seed || slot_count (u32 LE) || for each slot: scheduled_ms (u64 LE) || priority (u8) || intervention_id (16 bytes) )
```

### 5.2 Phi-derived timing computation

```rust
const GOLDENSECOND: u64 = 1618; // ms
const PHI_INV_NUM: u64 = 618_033_988_749_894; // scaled 1/phi * 10^15
const PHI_INV_DEN: u64 = 1_000_000_000_000_000;

fn next_window(last_ms: u64, depth: AttentionDepth) -> u64 {
    let n = depth as u64; // 1-5
    // interval = GOLDENSECOND * (PHI_INV)^n
    // computed with integer arithmetic to avoid FP nondeterminism
    let mut interval = GOLDENSECOND;
    for _ in 0..n {
        interval = (interval * PHI_INV_NUM) / PHI_INV_DEN;
    }
    last_ms + interval
}
```

### 5.3 Collision resistance
- `entropy` input (8 bytes) zapewnia, że nawet identyczne profile w tym samym milisecundzie produkują różne hashe
- `user_pseudonym` (UUIDv4) zapewnia unikalność per user
- Łączna przestrzeń entropii: 16 + 8 + 1 + 8 = 33 bytes → `2^264` możliwych inputów

### 10.3 Verification (runtime)
```typescript
import { verify } from '@noble/ed25519';

async function verifyWasmIntegrity(wasmBytes: Uint8Array, signature: Uint8Array): Promise<boolean> {
  const hash = await crypto.subtle.digest('SHA-256', wasmBytes);
  return verify(signature, new Uint8Array(hash), PUBLIC_KEY);
}
```

### 11.2 Equivalence runner (CI)

```typescript
// 05_apps/ci/equivalence-test/run.ts
interface EquivalenceResult {
  readonly nativeHash: string;
  readonly wasmHash: string;
  readonly cpuHash: string;
  readonly match: boolean;
  readonly latencyMs: { native: number; wasm: number; cpu: number };
}

async function runEquivalenceBatch(inputs: InterventionInput[]): Promise<EquivalenceResult[]> {
  const native = rustNative.computeBatch(inputs);
  const wasm = await wasmModule.computeBatch(inputs);
  const cpu = cpuFallback.computeBatch(inputs);
  
  return inputs.map((_, i) => ({
    nativeHash: native[i].outputHash,
    wasmHash: wasm[i].outputHash,
    cpuHash: cpu[i].outputHash,
    match: native[i].outputHash === wasm[i].outputHash && wasm[i].outputHash === cpu[i].outputHash,
    latencyMs: { native: native[i].latency, wasm: wasm[i].latency, cpu: cpu[i].latency },
  }));
}
```

### 11.3 Acceptance criteria
- **Determinism:** 100% match across all three targets for 10k identical inputs
- **Collision:** 0 collisions for 10k unique inputs (birthday bound: `Pr[collision] ≈ 10k² / 2^257 ≈ 0`)
- **Latency:** p99 < 382ms na wszystkich targetach
- **Divergence:** jakakolwiek divergencja → FAIL release, blokada BG3
